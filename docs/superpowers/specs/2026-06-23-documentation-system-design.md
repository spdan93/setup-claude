# Documentation System — Design Spec

- **Date**: 2026-06-23
- **Status**: approved (brainstorming) → ready for implementation plan
- **Scope**: the `.claude` kit (`setup-claude`)
- **Targets**: `commands/documentation.md`, `commands/ship.md`, `commands/commit.md`, new `skills/doc-*`, `CONVENTIONS.md`, docs.

## Summary

Overhaul documentation in the kit along two related axes:

- **Part A — Per-commit changelog**: every `/commit` writes a changelog artifact under
  `docs/changelog/` describing exactly what that commit changed (author, date/time, etc.),
  and commits it together with the change. Replaces the old "update CHANGELOG.md / docs-site
  page" model.
- **Part B — On-demand documentation system**: a set of **doc-type skills**, each carrying
  its own methodology + a `templates/` folder, generating one document on demand. A
  reworked `/documentation` command routes to the chosen type. Ships with 4 types in v1
  (technical, functional, test plan, API/Swagger) and is trivial to extend.

## Goals

- Generate, on demand, one well-formed document of a chosen type, grounded in the code/feature.
- Each doc type carries **methodology** (how to write it well), not just a format.
- Let the user **select a template** when more than one exists for a type; let them add their
  own templates by dropping files into the type's `templates/` folder.
- Make the changelog an automatic, granular, per-commit artifact with full authorship/time info.
- Stay project-agnostic; durable docs live at the repo root under `docs/` (versioned), not in `.claude`.

## Non-goals

- No "scaffold the whole corporate doc set at once" mode (usage is on-demand, per artifact).
- v1 does not ship non-functional (NFR) docs or use-case docs (framework supports adding them).
- No auto-generated rolling `CHANGELOG.md` aggregate (per-commit files are the source of truth;
  an optional index is the only aggregate).

## Decisions (locked in brainstorming)

1. Usage is **on-demand, per artifact** (not scaffolding).
2. Each doc type carries **methodology + template** → **skill per type**.
3. Changelog: **one file per commit, generated by `/commit`**.
4. v1 doc types: **technical, functional, test plan (caderno de testes), API/Swagger**.
5. **Template selection**: if the user doesn't specify a template, the skill lists the
   available templates for that type and asks the user to pick.

## Part A — Per-commit Changelog

### Trigger
`/commit` generates the changelog entry as part of its flow. `/ship` inherits it (it calls
`/commit`). No other command writes changelog entries automatically.

### Location & naming
`docs/changelog/YYYY_MM_DD-HHMM-{slug}.md`
- `slug` = kebab-case of the commit title (≤ 50 chars), same slug rule as `pipeline_id`.
- Time included so multiple commits the same day don't collide.

### Content (fields)
The entry mirrors the commit's 5-section body plus metadata:

```markdown
---
date: YYYY-MM-DD
time: HH:MM
author: <git config user.name>
branch: <current branch>
type: <feat|fix|refactor|...>
scope: <scope>
commit_title: <type(scope): title [ISSUE-ID?]>
files_changed: <N>
issue: <ISSUE-ID | null>
---

# <commit title>

## Cause
<why the change was necessary>

## Changes
<files / functions / components changed>

## Consequence
<impact / result>

## Functionality
<how it works>

## Gain
<technical or business benefit>

## Files
- path/a.ext
- path/b.ext
```

Labels follow the project's language convention (English default), consistent with `/commit`.

### Flow inside `/commit`
1. Analyze the diff and compose the 5-section message (existing behavior).
2. Resolve metadata: `git config user.name`, `date`, current branch, changed files, type/scope, issue (if linked).
3. Render `docs/changelog/YYYY_MM_DD-HHMM-{slug}.md` from the changelog template, filled with (2) + the 5 sections.
4. `git add -A` (now includes the changelog file) → create the commit → push.
   The changelog ships **inside** the commit it describes.

### Template
The entry format is itself a customizable template:
`skills/doc-changelog/templates/commit-entry.md` (placeholders for the fields above).
`/commit` reads it and fills it; if absent, falls back to the inline format above.

### Optional index
`docs/changelog/INDEX.md` — an append-only one-line-per-entry list (date · title · author),
updated by `/commit`. Optional; skip if it complicates the flow.

### `/ship` interaction
`/ship` **drops** its old changelog logic (CHANGELOG.md / docs-site page updates). New `/ship`:
`bump version → (optional) /documentation → /commit (writes the changelog) → push`.

## Part B — On-demand Documentation System

### Doc-type skills
One skill per type, grouped by a `doc-` prefix:

```
skills/doc-technical/SKILL.md      + templates/
skills/doc-functional/SKILL.md     + templates/
skills/doc-test-plan/SKILL.md      + templates/
skills/doc-api/SKILL.md            + templates/
skills/doc-changelog/templates/    (template only; format consumed by /commit)
```

### SKILL.md anatomy (per type)
- **Frontmatter**: `name`, `description` (phrased to auto-trigger, e.g. "Use when asked to
  document … / generate a test plan / produce API/Swagger docs").
- **Methodology**: what makes this doc type good — what to include, what to ask the user,
  common pitfalls, how to ground it in the codebase.
- **Process**:
  1. Gather context (read the relevant code/feature; for API, inspect routes/controllers; for
     test plan, read the Plan's TC-* if present).
  2. **Select a template** (algorithm below).
  3. Fill the template with grounded content.
  4. Write to the output location with the naming convention; report the path.
- **Quality gate**: a checklist specific to the type.

### Template selection algorithm
1. If the user passed `--template=<name>` → use `templates/<name>`.
2. Else list `templates/*` for the type. Each template's first line / frontmatter carries a
   one-line description.
   - 0 templates → error ("no templates for this type; add one to `skills/doc-<type>/templates/`").
   - 1 template → use it (no prompt).
   - ≥2 templates → `AskUserQuestion` showing each template (name + one-line description);
     use the chosen one.
3. The user adds custom templates simply by dropping files into `skills/doc-<type>/templates/`.

### `/documentation` router (reworked)
Replaces the current 4-mode command.
- `/documentation` (no args) → `AskUserQuestion` listing the available doc types (discovered
  from `skills/doc-*` that contain a `SKILL.md`; `doc-changelog` is template-only and is
  excluded — the changelog is automatic in `/commit`) → invokes `Skill("doc-<type>")`.
- `/documentation <type> [--template=…] [path/feature]` → routes directly to that type.
- The old modes are retired: "feature doc" → pick technical/functional; "changelog" → now
  automatic in `/commit`; "free command" → just talk to Claude.

### Output locations & naming
`docs/<category>/YYYY_MM_DD-{slug}.md` (slug rule = `pipeline_id`):

| Type | Category dir | Notes |
|------|-------------|-------|
| changelog | `docs/changelog/` | per-commit, `…-HHMM-{slug}.md` |
| technical | `docs/technical/` | |
| functional | `docs/functional/` | |
| test plan | `docs/test-plans/` | distinct from `docs/test-evidence/` (E2E run reports) |
| API/Swagger | `docs/api/` | markdown doc + `docs/api/openapi.yaml` spec |

All under the **repo root** `docs/` (versioned), coexisting with `docs/prds/`, `docs/plans/`,
`docs/test-evidence/`. Never inside `.claude/`.

### Starter templates (v1)
- `doc-technical/templates/`: `technical-design.md`, `adr.md` (architecture decision record)
- `doc-functional/templates/`: `functional-spec.md`, `business-rules.md`
- `doc-test-plan/templates/`: `test-plan.md` (scenarios/cases, linked to TC-* when available)
- `doc-api/templates/`: `openapi-swagger.yaml`, `rest-endpoints.md`

### Adding a new type (extensibility)
Create `skills/doc-<new>/SKILL.md` (methodology + process) and a `templates/` folder. It is
auto-discovered by `/documentation` (which lists `skills/doc-*`) and auto-triggers by its
description. Document this in CONVENTIONS.

## Out of scope / future

- Non-functional (NFR) and use-case doc types (add later via the extension path).
- Rolling aggregate CHANGELOG.md (only the optional `docs/changelog/INDEX.md`).
- Publishing docs to an external docs site.

## Open questions

None — resolved in brainstorming. The `docs/changelog/INDEX.md` aggregate is optional and may
be dropped during implementation if it adds friction.
